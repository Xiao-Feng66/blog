# 博客前台加载性能优化

> 状态：已完成

## 背景

博客文章页打开需等待 10+ 秒。根因有三层叠加：

1. **无静态生成**：`/posts/[slug]` 和 `/tags/[slug]` 没有 `generateStaticParams`，每次请求实时走 DB 查询 + MDX 编译 + Shiki 语法高亮
2. **Supabase 免费版冷启动**：项目 7 天无活跃后自动暂停，唤醒需 5–15 秒
3. **无连接池**：Prisma 在 Serverless 上每次调用新建 DB 连接，无复用

## 方案

### 页面层：SSG + ISR（1 天缓存）

所有前台页面加 `revalidate = 86400`（1 天）。文章页和标签页额外加 `generateStaticParams`，build 时预生成静态 HTML 推到 CDN。

| 页面 | 改动 |
|------|------|
| `/posts/[slug]` | `generateStaticParams` + `revalidate = 86400` |
| `/tags/[slug]` | `generateStaticParams` + `revalidate = 86400` |
| `/`（首页） | `revalidate = 86400` |
| `/posts`（文章列表） | `revalidate = 86400` |
| `/tags`（标签列表） | 移除 `force-dynamic`，改为 `revalidate = 86400` |

### API 层：写操作后全量清缓存

所有写操作（文章增删改、标签增删改）成功后，统一调 `revalidatePath('/', 'layout')` 清除全部前台页面缓存。不区分影响范围，全清一次，逻辑简单且正确。

涉及 4 个 API 路由文件：
- `POST /api/posts`（创建文章）
- `PUT/DELETE /api/posts/[id]`（更新/删除文章）
- `POST /api/tags`（创建标签）
- `PUT/DELETE /api/tags/[id]`（更新/删除标签）

### Supabase 连接池（手动配置，不涉及代码）

将 Vercel 环境变量 `DATABASE_URL` 改为 Supabase Pooler URL（Session mode）。路径：Supabase Dashboard → Project Settings → Database → Connection pooling。

## 变更评估

预估文件数：9 个（5 个页面 + 4 个 API 路由）
涉及模块：前台路由层、后台 API 层
影响范围：前台所有页面 + 后台写操作
风险：低——均为 Next.js 标准 API
